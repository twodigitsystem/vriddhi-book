### **PRD: Advanced Unit Management for Inventory Items**

### 1\. Introduction & Overview 📝

Many retail businesses, especially in pharmacy and grocery, purchase products in bulk units (e.g., a **Box** of 100 tablets) but sell them in smaller, individual units (e.g., a single **Piece** or **Tablet**). Our current inventory system lacks the granularity to handle this, leading to inaccurate stock tracking and manual workarounds.

This document outlines the requirements for an **Advanced Unit Management** feature. It will be implemented as a modal UI within the "Add/Edit Item" form, allowing users to define a **Base Unit**, a **Secondary Unit**, and the **Conversion Rate** between them. This will enable precise, automated inventory control and flexible pricing.

### 2\. User Stories

As a **Pharmacy Owner/Inventory Manager**, I want to:

1.  **Define multiple units for a single product** (e.g., a "Strip" containing 10 "Tablets") so that my inventory is tracked accurately whether I sell a full strip or a single loose tablet.
2.  **Set a clear conversion rate** between the units so the system automatically knows how many secondary units are contained within a base unit.
3.  **Be guided through the process with a simple UI** so that I can set up complex products quickly and without errors.
4.  **Be prevented from saving incomplete or invalid unit data** so that data integrity is maintained across the application.

### 3\. Feature Requirements 📋

#### **Functional Requirements (FR)**

  * **FR1: Modal Trigger & Context**

      * On the "Add Item" and "Edit Item" pages, there will be a button labeled **"Set Units"** next to the primary unit field.
      * Clicking this button will launch the "Select Unit" modal.
      * The modal will receive the product's current unit configuration, if one exists, to pre-fill the form.

  * **FR2: UI Components (Shadcn UI)**

      * **Modal:** The main container will be a `Dialog` component from Shadcn UI.
      * **Unit Selectors:** Both "BASE UNIT" and "SECONDARY UNIT" fields will use the Shadcn UI `Combobox` (a searchable `Select`) component to allow users to easily find units from a predefined list.
      * **Conversion Input:** The conversion rate will be defined using a `CheckBox` with two options. Each option will contain a dynamic label and a number `Input` component.
      * **Action Buttons:** The modal footer will contain a primary "Save" `Button` and a secondary "Cancel" `Button`.

  * **FR3: Dynamic UI Behavior (React)**

      * The labels in the "Conversion Rates" section must dynamically update based on the selections in the "BASE UNIT" and "SECONDARY UNIT" dropdowns. (e.g., "1 BOX = [\_] PIECES").
      * The "Save" button shall be **disabled** by default and only become enabled when the form is valid (base unit selected and seconadary unit is optional, if selected then conversion rate will show).
      * The input field for the un-selected checkbox option must be disabled.

  * **FR4: Form Handling & Validation (React Hook Form & Zod)**

      * The modal's form state will be managed by **React Hook Form**.
      * Client-side and server-side validation will be handled by a **Zod** schema.
      * The schema will enforce:
          * `baseUnitId`: Must be a valid CUID string.
          * `secondaryUnitId`: Must be a valid CUID string (and cannot be the same as `baseUnitId`).
          * `conversionRate`: Must be a positive number greater than 0.

  * **FR5: Data Persistence (PostgreSQL, Prisma, Server Actions)**

      * Upon successful form submission, a **Next.js Server Action** will be invoked.
      * This action will receive the validated form data.
      * It will use **Prisma Client** to update the corresponding `Product` record in the PostgreSQL database.
      * After the database update, the client-side data will be refreshed using **Tanstack Query's** `queryClient.invalidateQueries` to ensure the UI reflects the new state.


 **Data Fetching:** The list of available units will be fetched from the database using a Server Component or `useQuery` (Tanstack Query) and passed down to the form.



### 5\. Acceptance Criteria ✅

**GIVEN** a user is on the "Add/Edit Item" page:

  * **WHEN** the user clicks the "Set Units" button, **THEN** the "Select Unit" modal must appear.
  * **GIVEN** the modal is open, **WHEN** the user selects "BOX" as the Base Unit and "PIECES" as the Secondary Unit, **THEN** the conversion rate label must read "1 BOX = [\_] PIECES".
  * **GIVEN** the form is incomplete, **WHEN** the user attempts to click "Save", **THEN** the button must be disabled or show a validation error.
  * **GIVEN** the user has filled out the form correctly (e.g., `1 BOX = 10 PIECES`), **WHEN** they click "Save", **THEN** the Server Action must be called, the data must be saved to the database, the modal must close
  * **GIVEN** the modal is open, **WHEN** the user clicks "Cancel" or presses the `Esc` key, **THEN** the modal must close without saving any changes.
